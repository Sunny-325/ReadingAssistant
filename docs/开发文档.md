# 中文文本阅读障碍辅助工具 - 开发文档

## API文档

### 1. 文本处理相关API

#### POST /api/text/process
- **功能**：文本处理，实现12项核心阅读辅助功能
- **请求体**：
  ```json
  {
    "text": "要处理的文本",
    "options": {
      "enableChunk": true,  // 启用意群划分
      "chunkSize": 5,  // 意群大小
      "enableMainContent": true,  // 启用主次内容区分
      "enableSimplify": true,  // 启用文本简化
      "pos_tagging": true  // 启用词性标注
    }
  }
  ```
- **响应**：
  ```json
  {
    "original_text": "原始文本",
    "processed_text": "处理后的文本",
    "simplified_text": "简化后的文本",
    "segments": [
      {
        "id": 1,
        "text": "意群1",
        "start_pos": 0,
        "end_pos": 10,
        "importance": 0.8,
        "is_primary": true
      }
    ],
    "simplified_segments": [
      {
        "id": 1,
        "text": "简化意群1",
        "start_pos": 0,
        "end_pos": 8,
        "importance": 0.8,
        "is_primary": true
      }
    ],
    "pos_tags": [
      {
        "word": "词语",
        "pos": "n",
        "pos_name": "名词",
        "start_pos": 0,
        "end_pos": 2
      }
    ],
    "simplified_pos_tags": [
      {
        "word": "简化",
        "pos": "v",
        "pos_name": "动词",
        "start_pos": 0,
        "end_pos": 2
      }
    ],
    "metadata": {
      "is_long_text": false,
      "chunk_count": 0
    }
  }
  ```

#### GET /api/text/definition/{word}
- **功能**：词语释义查询
- **路径参数**：
  - `word`：要查询的词语
- **查询参数**：
  - `context`：可选，词语的上下文
- **响应**：
  ```json
  {
    "word": "查询的词语",
    "phonetic": "读音",
    "definitions": ["释义1", "释义2"],
    "examples": ["例句1", "例句2"],
    "raw_response": "详细说明"
  }
  ```

### 2. 文件相关API

#### POST /api/files/upload
- **功能**：文件上传
- **请求体**：
  - `file`：要上传的文件（multipart/form-data格式）
- **响应**：
  ```json
  {
    "success": true,
    "text": "文件内容",
    "metadata": {
      "file_name": "文件名",
      "file_size": 1024,  // 文件大小（字节）
      "file_ext": "txt",  // 文件扩展名
      "encoding": "utf-8"  // 文件编码
    }
  }
  ```

#### GET /api/files/export
- **功能**：文件导出
- **查询参数**：
  - `text`：要导出的文本
  - `format`：导出格式（txt或docx）
- **响应**：
  ```json
  {
    "success": true,
    "file_path": "导出文件路径"
  }
  ```

#### GET /api/files/list
- **功能**：获取文件列表
- **响应**：
  ```json
  {
    "success": true,
    "files": [
      {
        "file_name": "文件名1",
        "file_size": 1024,
        "create_time": "2023-01-01T12:00:00"
      },
      {
        "file_name": "文件名2",
        "file_size": 2048,
        "create_time": "2023-01-02T12:00:00"
      }
    ]
  }
  ```

### 3. 系统相关API

#### GET /api/system/status
- **功能**：获取系统状态
- **响应**：
  ```json
  {
    "success": true,
    "memory_usage": {
      "cpu_used": 1024,  // CPU内存使用（MB）
      "cpu_total": 8192,  // 总CPU内存（MB）
      "gpu_used": 1536,  // GPU显存使用（MB）
      "gpu_total": 4096  // 总GPU显存（MB）
    },
    "model_loaded": true,  // 模型是否已加载
    "device": "cuda"  // 当前使用的设备
  }
  ```

#### GET /api/system/health
- **功能**：健康检查
- **响应**：
  ```json
  {
    "success": true,
    "status": "healthy",
    "timestamp": "2023-01-01T12:00:00"
  }
  ```

#### POST /api/system/cleanup
- **功能**：清理内存
- **响应**：
  ```json
  {
    "success": true,
    "message": "内存清理完成",
    "memory_released": 512  // 释放的内存（MB）
  }
  ```

## 代码规范

### 1. 后端代码规范

#### Python代码规范
- 遵循PEP 8编码规范
- 使用4个空格缩进，禁止使用制表符
- 每行代码长度不超过100个字符
- 函数和方法使用小写字母加下划线的命名方式（snake_case）
- 类使用首字母大写的驼峰命名方式（CamelCase）
- 导入顺序：标准库、第三方库、本地模块
- 函数和方法必须添加文档字符串，使用Google风格
- 变量命名要清晰，避免使用单个字母（循环变量除外）

#### FastAPI规范
- 使用路由前缀（/api）
- 使用请求和响应模型（Pydantic）
- 错误处理使用HTTPException
- 日志记录使用Python内置的logging模块
- API文档使用Swagger UI（自动生成）

### 2. 前端代码规范

#### Vue代码规范
- 遵循Vue 3 Composition API
- 组件命名使用首字母大写的驼峰命名方式
- 使用单文件组件（.vue）
- 脚本部分使用setup语法糖
- 模板部分使用2个空格缩进
- CSS使用scoped属性，避免样式冲突
- 状态管理使用Pinia

#### JavaScript/TypeScript规范
- 变量和函数使用小写字母加下划线的命名方式（snake_case）
- 类使用首字母大写的驼峰命名方式（CamelCase）
- 每行代码长度不超过100个字符
- 使用const和let，禁止使用var
- 箭头函数优先于普通函数
- 模板字符串优先于字符串拼接

## 部署指南

### 1. 环境要求

#### 硬件要求
- CPU：至少4核心
- 内存：至少8GB
- GPU：至少4GB显存（推荐）
- 存储空间：至少50GB

#### 软件要求
- 操作系统：Windows 10/11或Linux（Ubuntu 20.04+）
- Python 3.8+ 
- Node.js 16+ 
- Docker（可选，用于容器化部署）
- MySQL 8.0+ 

### 2. 本地部署

#### 后端部署
1. 进入backend目录
2. 安装依赖：`pip install -r requirements.txt`
3. 初始化数据库：`python init_database.py`
4. 启动后端服务：`uvicorn main:app --reload --host 0.0.0.0 --port 8000`

#### 前端部署
1. 进入frontend目录
2. 安装依赖：`npm install`
3. 启动前端服务：`npm run dev`
4. 在浏览器中访问：http://localhost:5173

### 3. Docker部署

#### 构建镜像
1. 进入backend目录
2. 构建后端镜像：`docker build -t reading-assistant-backend .`
3. 进入frontend目录
4. 构建前端镜像：`docker build -t reading-assistant-frontend .`

#### 使用Docker Compose部署
1. 进入backend目录
2. 启动服务：`docker-compose up -d`
3. 访问应用：http://localhost:80

### 4. 生产环境部署

#### 后端配置
- 使用Gunicorn作为WSGI服务器
- 配置Nginx作为反向代理
- 设置环境变量：
  - `ENVIRONMENT=production`
  - `DATABASE_URL=mysql+pymysql://user:password@host:port/database`
  - `SECRET_KEY=your_secret_key`

#### 前端配置
- 构建生产版本：`npm run build`
- 将dist目录部署到Nginx或其他静态文件服务器
- 配置Nginx反向代理API请求

### 5. 模型部署

#### 模型下载与安装
- 安装Ollama：访问 https://ollama.com/download 下载并安装
- 下载Qwen2.5-3B模型：`ollama pull qwen2.5:3b-instruct-q4_K_M`
- 模型会自动存储在Ollama的默认目录中，无需手动移动文件

#### 模型优化
- 使用4位量化版本（q4_K_M），显存占用约2GB
- 通过Ollama自动管理模型内存和设备分配
- 利用Ollama的流式推理能力，提高响应速度
- 实现模型状态检查，确保服务可用性

## 开发流程

### 1. 分支管理
- `main`：主分支，用于发布生产版本
- `develop`：开发分支，用于集成所有功能分支
- `feature/xxx`：功能分支，用于开发新功能
- `bugfix/xxx`：bug修复分支，用于修复bug

### 2. 开发步骤
1. 从develop分支创建功能分支
2. 实现功能，编写测试
3. 提交代码，创建Pull Request
4. 代码审查
5. 合并到develop分支
6. 发布前合并到main分支

### 3. 测试流程
- 单元测试：使用pytest测试单个功能模块
- 集成测试：测试多个模块的协同工作
- 端到端测试：测试完整的业务流程
- 性能测试：测试系统的响应速度和资源占用

### 4. 代码审查
- 至少两名开发人员参与审查
- 检查代码规范和质量
- 检查功能实现是否正确
- 检查是否存在安全漏洞
- 检查测试覆盖率

## 安全规范

### 1. 数据安全
- 敏感数据加密存储（如密码使用bcrypt加密）
- 数据库连接使用环境变量配置
- 避免在日志中记录敏感信息

### 2. API安全
- 使用HTTPS协议
- 实现请求速率限制
- 验证请求参数
- 处理异常情况，避免泄露系统信息

### 3. 模型安全
- 限制模型的访问权限
- 监控模型的使用情况
- 定期更新模型，修复安全漏洞

### 4. 部署安全
- 定期更新系统和依赖包
- 配置防火墙，限制访问端口
- 监控系统日志，及时发现异常情况
